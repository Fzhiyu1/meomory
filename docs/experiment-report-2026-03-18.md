## 二、噪声鲁棒性分析

### 2.1 三代算法对比（LoCoMo 全量 1976 题, 10 轮）

模拟不同反馈准确率下的表现（noise_accuracy < 1.0 时以该概率给 GT，否则随机片段）：

| 反馈准确率 | gen036 (v1 简单函数) | gen057 (中期 class) | gen204 (最强 class) |
|:---:|:---:|:---:|:---:|
| 100% GT | 27.5% | 50.1% | **60.1%** |
| 90% | 26.5% (-1.0) | **47.8%** (-2.3) | 54.3% (-5.8) |
| 80% | 25.7% (-1.8) | **46.3%** (-3.8) | 42.0% (-18.1) |
| 70% | 24.8% (-2.7) | **40.6%** (-9.5) | 36.9% (-23.2) |
| 60% | 23.7% (-3.8) | **36.7%** (-13.4) | 23.7% (-36.4) |
| 50% (纯猜) | 22.2% (-5.3) | **30.7%** (-19.4) | 16.9% (-43.2) |

### 2.2 关键发现

1. **精度-鲁棒性 trade-off 确认**：gen204 在 clean GT 下最强（60.1%），但噪声敏感度远高于 gen057
2. **80% 反馈是交叉点**：gen057 (46.3%) 反超 gen204 (42.0%)，70% 以下 gen057 全面碾压
3. **gen204 在 60% 噪声下退化到 cosine baseline**（23.7% vs cosine 22.4%），gen057 仍有 36.7%
4. **gen036 极其鲁棒但极其弱**：100→50% 只掉 5.3pp，但 100% GT 也才 27.5%——没学到什么所以噪声也破坏不了什么
5. **进化的 Adam/Momentum 机制在 clean 下学得更激进，但对错误信号也更敏感**

### 2.3 对比之前 500 题子集结果

gen057 之前在 500 题子集上的噪声测试（实验报告 2026-03-17 第 11 章）：

| 反馈准确率 | 500 题 (旧) | 1976 题 (新) |
|:---:|:---:|:---:|
| 100% | 87.4% | 50.1% |
| 90% | 87.0% (-0.4pp) | 47.8% (-2.3pp) |
| 80% | 86.2% (-1.2pp) | 46.3% (-3.8pp) |

500 题成绩远高于全量（87.4% vs 50.1%），证实了子集过拟合效应。但**相对鲁棒性保持一致**：gen057 在两个尺度上都表现出良好的噪声容忍度。

### 2.4 部署场景推荐

| 场景 | 推荐算法 | 理由 |
|------|---------|------|
| GT 反馈可用（人工标注） | gen204 | 60.1% 最高 |
| LLM Judge ~90% 准确率 | gen204 | 54.3% 仍领先 |
| LLM Judge ~80% 准确率 | **gen057** | 46.3% > 42.0% |
| 低质量反馈 <70% | **gen057** | 鲁棒性远优于 gen204 |

## 四、FunSearch v5 框架升级

### 4.1 改动清单

| 改动 | 文件 | 效果 |
|------|------|------|
| 并行 LLM 采样 | runner.py | `asyncio.gather` 6 个 sample 并行，采样时间 6×T → ~T |
| 混合 fitness | runner.py + evaluator.py | 70% clean + 30% noisy@80%，同时优化精度和鲁棒性 |
| 异步全量验证 | runner.py | NEW BEST 触发后台全评，不阻塞主循环 |
| noise_accuracy 参数 | evaluator.py | 支持任意噪声级别的评测 |

### 4.2 混合 fitness 设计

```
mixed_score = 0.7 × P@1(100% GT) + 0.3 × P@1(80% GT)
```

- 权重 70/30 确保精度仍是主要选择压力
- 80% 噪声级别模拟真实 LLM Judge 场景（Judge prompt 进化到 90.9%，但实际部署时更保守估计 80%）
- 噪声使用 `random.Random(42 + offset)` 确保可复现

### 4.3 预期效果

基于 gen057 vs gen204 的数据，混合 fitness 应该能逼进化出：
- clean GT 下接近 gen204 水平（~55-60%）
- 80% 噪声下接近 gen057 水平（~45%）
- 而不是像 gen204 那样 80% 噪声下崩到 42%

### 5.1 本机全量验证

|:---:|:---:|:---:|

| P@3 | — | 74.0% |
| P@5 | — | 80.7% |
| 耗时 | 146s | 895s |

膨胀比 87.0% / 56.6% = 1.54x。

### 5.2 噪声鲁棒性（完整结果）

| 反馈准确率 | gen036 (v1) | gen057 (中期) | **gen059 (新王)** | gen204 (旧最强) |
|:---:|:---:|:---:|:---:|:---:|
| 100% GT | 27.5% | 50.1% | **56.6%** | **60.1%** |
| 90% | 26.5% | 47.8% | **53.2%** | 54.3% |
| 80% | 25.7% | 46.3% | **51.7%** | 42.0% |
| 70% | 24.8% | 40.6% | **46.3%** | 36.9% |
| 60% | 23.7% | 36.7% | **42.9%** | 23.7% |
| 50% (纯猜) | 22.2% | 30.7% | **35.8%** | 16.9% |

**gen059 在除 100% GT 外的所有噪声级别全面碾压。** 总衰减（100%→50%）：gen059 -20.8pp vs gen204 -43.2pp，鲁棒性差 2 倍。

### 5.3 架构对比

| 特性 | gen057 | gen059 |
|------|:---:|:---:|
| 权重矩阵 | 1 个 (M) | **2 个 (M + M2)** |
| M2 处理 abs(x) | 无 | **有（类 ReLU 特征）** |
| 通道混合 mix[] | 无 | **可学习，per-dim 0~0.5** |
| 时序关联 | 无 | **last_key/target/pred 差分** |
| 代码量 | 5KB | 8KB (+60%) |
| 单次操作延迟 | ~15ms | ~45ms |

gen059 的双通道提供冗余（一个通道被噪声干扰，另一个还在），时序关联平滑了单次错误反馈。**同时做到更准和更鲁棒。**

### 5.4 部署推荐更新

| 场景 | 推荐算法 | 理由 |
|------|---------|------|
| GT 反馈可用 | gen204 | 60.1% 最高 |
| LLM Judge ~80-90% | **gen059** | 51.7-53.2%，综合最强 |
| 极低质量反馈 <70% | gen057 | 相对衰减最小 |

## 七、泛化危机与 v6 重新设计

### 7.1 冷启动实验——发现灾难性干扰

Train 500 题 GT 反馈，Test 1476 题评测（DGD 从未见过）：

| | Train (500) | Test (1476) |
|--|:-:|:-:|
| Cosine baseline | 26.0% | **21.2%** |
| gen005 (v6 online) 10 轮 | 27.4% | **1.4%** |
| gen059 (v4 overfit) 10 轮 | 72.6% | **0.9%** |

**结论：所有进化出的算法在未见数据上比 cosine 更差。** 全局 M 矩阵更新导致灾难性干扰——学了 A 会破坏 B。之前 60.1% 的成绩完全因为 train=test。

### 7.2 v6 设计——在线流式评测 + 打乱顺序

**核心改动：**
- 评测改为"先测再学"（每题先评分再 GT 反馈）
- 题目顺序打乱（跨对话交叉，暴露干扰）
- 算法自己控制超参（evaluator 只传 dim）
- 5 种架构 seed：原始 DGD / CMS 三层 / gen059 / gen057 / blank

**打乱后 seed 基线：**

| Seed | 在线 P@1 (打乱) |
|------|:-:|
| blank (不学习) | **22.2%** |
| CMS 三层 | 16.0% |
| gen059 双通道 | 12.2% |
| gen057 单通道 | 12.7% |
| 原始 DGD | 0.6% |

学习型算法全部比 cosine 差——进化必须发明"学了不变差"的算法。

### 7.3 v6 早期进化发现

**gen011-island4-005 (25.0%)** — 从 blank seed 进化，3 个抗干扰机制：
1. **单位矩阵初始化 + 行归一化**：M 不偏离太远
2. **弹出回训**：旧样本弹出前用 10% 学习率再学一遍
3. **tanh 饱和输出**：防止某方向爆炸

**gen005-island0-001 (33.8%, 打乱前)** — 从原始 DGD 进化：
- RMSProp 自适应学习率
- 历史缓冲（20 条）做信心放大
- 被弹出时无回训

### 7.4 进化出的算法特征

v4/v5 进化出了"擅长记忆"的算法（Adam + Momentum + 双通道）。v6 在进化"擅长在线学习"的算法——完全不同的方向：

| 特征 | v4/v5 风格 | v6 风格 |
|------|-----------|--------|
| 目标 | 记住所有题 | 学了不破坏 |
| M 初始化 | 零矩阵 | 单位矩阵 |
| 更新 | 激进（高 eta） | 保守（正则化） |
| 防干扰 | 无 | 行归一化 + 弹出回训 |
| 历史缓冲 | 无 | 有（10-20 条） |

## 十一、v6 进化收敛与天花板确认（3月19日）

### 11.1 gen107 冷启动实验

v6 最强算法 gen107（经验回放 150 条）的冷启动测试：

| | Train (500) | Test (1476) |
|--|:-:|:-:|
| Cosine baseline | 26.0% | **21.2%** |
| gen107 1 轮 | 50.6% | 17.5% |
| gen107 5 轮 | 89.2% | 12.5% |
| gen107 10 轮 | 95.2% | **8.5%** |

经验回放把 Test 从 0.9%（gen059）改善到 17.5%（1 轮），但仍低于 cosine baseline 21.2%。训练越多 Test 越差——共享矩阵干扰无法消除。

### 11.2 进化收敛

- v6 跑了 370+ 轮（9b 30% + DeepSeek 20% + Opus 4.6 50%）
- 5 个岛（DGD/CMS/gen059/gen057/blank）全部趋同到 25-26%
- 打乱顺序后最高 26.0%（cosine baseline 22.4%，+3.6pp）
- 多条不同路径（行归一化 vs 残差连接 vs 经验回放）达到相同天花板
- **结论：共享 256×256 矩阵在打乱在线模式下的天花板约 26%**

### 11.3 Opus 4.6 表现

- 接入 Anthropic API（code.newcli.com），2 次重试 + DeepSeek 降级
- 193+ 次调用 **0 失败**
- 所有 NEW BEST 都由 Opus 产出
- 代码质量显著优于 9b 和 DeepSeek

### 11.4 进化发现的算法机制

| 机制 | 发现者 | 已知对应 | 效果 |
|------|--------|---------|------|
| 经验回放 | gen107 (Opus) | DQN (DeepMind 2015) | 0.9% → 17.5% |
| 行归一化 | gen011 (9b) | Weight Normalization | 限制 M 偏离 |
| 残差连接 | gen073 (Opus) | ResNet (2015) | 保留原始信号 |
| 弹出回训 | gen011 (9b) | 无直接对应 | 防止弹出时遗忘 |
| 误差归一化 | gen073 (Opus) | Gradient Clipping | 防止大误差暴力修改 |
| 学习率衰减 | gen107 (Opus) | Learning Rate Scheduling | 后期更保守 |

### 11.5 核心教训

1. **适应度幻觉**：train=test 产出 60.1%（假），在线流式产出 26%（真）
2. **研究者过拟合**：我自己也被 87% 锚定，博客和 README 中的对比数据基于错误评测
3. **共享矩阵的根本限制**：不是算法问题，是架构问题。进化在正确的 fitness 下仍然只能到 26%
4. **DGD 的真实定位**："重复模式加速器"——对反复出现的查询有效，对新查询无效
5. **慢和难是好信号**：v4 两小时 87%（假），v6 一整晚 26%（真）
