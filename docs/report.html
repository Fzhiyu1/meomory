关键发现

发现 1：DGD 在 BigBang 和 TheOffice 上显著提升

BigBang：DGD+Judge P@1=78.6%，比 cosine +21.5pp，追平 Ground Truth。
TheOffice：DGD+Judge P@1=50.0%，比 cosine +25.0pp，也追平 Ground Truth。
这两个数据集上 DGD 的效果一致且显著。

发现 2：DGD 在 Friends 上没有提升

Friends 的 DGD+Judge P@1=25.0%，和 cosine baseline 完全一样。Ground Truth 也只到 41.7%——说明这个数据集本身很难（答案片段和问题的语义相似度低，即使给正确反馈也只能提升到 41.7%）。
结论：DGD 不是万能的。当问题和答案的语义鸿沟太大时，M 矩阵的线性变换不足以弥补。

发现 3：CMS 三层在 280 次查询下无优势

CMS（三层 DGD）在 BigBang 上甚至比单层 DGD 更差（42.9% vs 78.6%）。原因：20 轮 × 14 题 = 280 次更新，不足以让三层分化出不同的时间尺度。高频层的快速遗忘反而冲掉了有效关联。
结论：CMS 需要更长的使用历史（几千次查询）才能体现价值。在短期 benchmark 上不适用。

发现 4：9B 小模型 ≈ DeepSeek 大模型

在 BigBang 上 qwen3.5:9b (78.6%) 甚至略优于 DeepSeek (71.4%)。TheOffice 上两者持平 (50.0%)。Friends 上两者都失败。
结论：Judge 角色不需要大模型。整个系统可以完全离线运行在消费级 GPU 上。

DGD 如何工作

DGD 不替代向量库，而是套在向量库前面。查询向量在进入向量库之前，先经过一个 256×256 的权重矩阵 M 变换。

M 做的事：把查询向量旋转、拉伸，让它更靠近正确答案的方向。向量库里的数据不变，变的是你看向量库的角度。

M 通过 LLM Judge 的自动反馈学习：agent 用了哪条记忆 → 强化那条关联。65536 个参数，一次矩阵乘法完成。

普通 RAG：query → embedding → 直接比较 → 返回
meomory：query → embedding → M × query → 比较 → 返回 → 观察 → 更新 M

完整消息流

用户消息 → DGD 匹配记忆 → 注入上下文 → LLM 回复 ↓ LLM Judge 判断引用了哪些 ↓ DGD 强化被引用的关联 → M 演化

局限性

• 34 题仍然偏少——但比最初的 14 题好了一倍多

• Friends 上 DGD 完全无效——需要分析原因（语义鸿沟？数据特性？）

• CMS 在短期 benchmark 上反而有害——需要更长的使用历史验证

• 未部署到真实 agent 场景

• 未对比 RAG + Reranker baseline

结论

DGD + LLM Judge 在 2/3 数据集上追平了 Ground Truth 上界（BigBang +21.5pp, TheOffice +25pp），全自动无需人工标注。9B 小模型足以做 Judge。

但 DGD 不是万能的——在 Friends 上完全无效，CMS 在短期实验中反而有害。方向可行，但不是所有场景都适用。
